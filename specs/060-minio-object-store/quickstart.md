# Quickstart — MinIO Object Storage

**Feature**: 060-minio-object-store

This quickstart walks through (a) deploying MinIO via the standard vanlab GitOps flow, (b) running the one-time `mc` bootstrap to provision buckets and the Longhorn-scoped user, and (c) verifying the result. It is intended to be the runbook the implementer follows when executing tasks generated by `/speckit.tasks`.

## Prerequisites

- ArgoCD, Sealed Secrets controller, Longhorn, Traefik, cert-manager all healthy.
- fleet1.lan wildcard cert (spec 054) present in cluster.
- `mc` (MinIO CLI) installed on the management laptop.
- `kubectl` configured for the cluster.

## Step 1 — Add MinIO secrets to `group_vars/all.yml`

Generate four values (use `pwgen -s 32 1` or similar):

```yaml
# group_vars/all.yml  (NOT committed — encrypted source-of-truth)
minio_root_user: vanlab-admin
minio_root_password: "<32+ random chars>"

longhorn_minio_access_key: "<20 random alphanumeric chars>"
longhorn_minio_secret_key: "<40 random alphanumeric chars>"
```

Update `group_vars/example.all.yml` with placeholder keys (no real values) and commit *that* file.

## Step 2 — Generate the SealedSecret

```bash
ansible-playbook playbooks/utilities/seal-secrets.yml
```

This extends `seal-secrets.yml` to produce `manifests/minio/prereqs/sealed-secrets.yaml`. Inspect the output — it should contain two `SealedSecret` resources (`minio-root` and `longhorn-minio-credentials-source`) and zero plaintext values.

## Step 3 — Commit and push

```bash
git checkout 060-minio-object-store
git add manifests/minio/ group_vars/example.all.yml playbooks/utilities/seal-secrets.yml
git commit -m "feat(minio): add MinIO manifests and sealed secrets"
git push gitea 060-minio-object-store
git push origin 060-minio-object-store
```

Merge via Gitea PR per the CLAUDE.md workflow.

## Step 4 — Register and sync via ArgoCD

```yaml
# group_vars/all.yml — argocd_apps additions
argocd_apps:
  # ...existing entries...
  - name: minio-prereqs
    path: manifests/minio/prereqs
  - name: minio
    path: manifests/minio/apps
```

```bash
ansible-playbook playbooks/cluster/services-deploy.yml --tags argocd-bootstrap
```

Watch `https://argocd.fleet1.cloud` → both `minio-prereqs` and `minio` Applications should reach Synced + Healthy within ~3 minutes.

## Step 5 — Verify the deployment

```bash
kubectl get pods -n minio
# Expected: 1 minio-* pod Ready 1/1

kubectl get pvc -n minio
# Expected: 200Gi PVC Bound, StorageClass: longhorn

kubectl get svc -n minio
# Expected: minio (port 9000), minio-console (port 9001)

kubectl get ingressroute -n minio
# Expected: minio-console with host minio.fleet1.lan

curl -s -o /dev/null -w "%{http_code}\n" \
  https://minio.fleet1.lan/login
# Expected: 200 (admin console UI loads over TLS)
```

Verify arm64 image was pulled:

```bash
kubectl get pod -n minio -o jsonpath='{.items[0].spec.containers[0].image}'
# Verify against the multi-arch manifest digest for arm64.
```

## Step 6 — Bootstrap buckets and the Longhorn user (one-time `mc` operations)

Fetch credentials from the cluster:

```bash
ROOT_USER=$(kubectl get secret minio-root -n minio -o jsonpath='{.data.rootUser}' | base64 -d)
ROOT_PASS=$(kubectl get secret minio-root -n minio -o jsonpath='{.data.rootPassword}' | base64 -d)
LH_AK=$(kubectl get secret longhorn-minio-credentials-source -n minio -o jsonpath='{.data.accessKey}' | base64 -d)
LH_SK=$(kubectl get secret longhorn-minio-credentials-source -n minio -o jsonpath='{.data.secretKey}' | base64 -d)
```

Open a port-forward (foreground or background):

```bash
kubectl port-forward -n minio svc/minio 9000:9000 &
PF_PID=$!
```

Configure `mc` and provision:

```bash
mc alias set vanlab-minio http://localhost:9000 "$ROOT_USER" "$ROOT_PASS"

mc mb --ignore-existing vanlab-minio/longhorn-backups
mc mb --ignore-existing vanlab-minio/vanlab-archive

# Create the scoped policy (write JSON to a temp file first)
cat > /tmp/longhorn-backups-rw.json <<'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    { "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject", "s3:DeleteObject"],
      "Resource": ["arn:aws:s3:::longhorn-backups/*"] },
    { "Effect": "Allow",
      "Action": ["s3:ListBucket", "s3:GetBucketLocation"],
      "Resource": ["arn:aws:s3:::longhorn-backups"] }
  ]
}
EOF
mc admin policy create vanlab-minio longhorn-backups-rw /tmp/longhorn-backups-rw.json

# Create the Longhorn user
mc admin user add vanlab-minio "$LH_AK" "$LH_SK"
mc admin policy attach vanlab-minio longhorn-backups-rw --user "$LH_AK"

# Tear down the port-forward
kill $PF_PID
```

## Step 7 — Verify scoped access (negative test)

This is the critical scope-enforcement check from FR-005.

```bash
kubectl port-forward -n minio svc/minio 9000:9000 &
PF_PID=$!

# Configure mc with the LONGHORN credential, not root
mc alias set vanlab-minio-longhorn http://localhost:9000 "$LH_AK" "$LH_SK"

# Positive: can list the assigned bucket
mc ls vanlab-minio-longhorn/longhorn-backups
# Expected: empty listing, exit 0

# Positive: can write to the assigned bucket
echo "hello" | mc pipe vanlab-minio-longhorn/longhorn-backups/test-object.txt
mc ls vanlab-minio-longhorn/longhorn-backups
# Expected: test-object.txt visible
mc rm vanlab-minio-longhorn/longhorn-backups/test-object.txt

# Negative: cannot list the other bucket
mc ls vanlab-minio-longhorn/vanlab-archive
# Expected: 403 / Access Denied. If this succeeds, the policy is too broad — STOP and fix.

# Negative: cannot list the bucket list itself
mc ls vanlab-minio-longhorn
# Expected: empty or 403 (depends on mc version)

kill $PF_PID
```

If any of the negative tests succeed (i.e., the Longhorn credential can read another bucket), the standup is incorrect — the bug is in the policy JSON or the `policy attach` step. Roll back and retry before proceeding to spec 061.

## Step 8 — Idempotency check

Re-run ArgoCD sync (or wait for the next auto-sync) — both Applications should remain Synced + Healthy with zero resource changes.

Re-run the `mc mb --ignore-existing` and `mc admin user add` commands from Step 6 — all should be no-ops (the chart's `mc mb --ignore-existing` is upsert-safe; `mc admin user add` with same credentials is idempotent).

## Step 9 — Hand-off to spec 061

When all checks pass, spec 061 (Longhorn backup target) can begin. It will:

- Re-seal the Longhorn access key + secret key into `manifests/longhorn-backup/prereqs/sealed-secrets.yaml` (namespace-scoped to `longhorn-system`).
- Configure Longhorn `BackupTarget` per the contract in `contracts/longhorn-backup-target.md`.
- Apply `RecurringJob` policies to critical PVCs.

## Rollback

If 060 needs to be rolled back:

1. ArgoCD UI → `minio` Application → Disable Auto-Sync → Delete (with `cascade`).
2. ArgoCD UI → `minio-prereqs` Application → Delete (with `cascade`).
3. `kubectl delete namespace minio`.
4. Manually clean up the Longhorn PVC if it survived (`kubectl get pv | grep minio` → `kubectl delete pv ...`).
5. Revert the commits in Gitea; pull main locally; remove `minio-prereqs` and `minio` from `argocd_apps`.

**Data loss warning**: deleting the namespace destroys the MinIO PVC and any data in it. Since 060 hasn't yet been the target of any Longhorn backup (that's spec 061's first sync), nothing of recoverable value is lost on a 060 rollback.
