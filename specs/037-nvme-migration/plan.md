# Implementation Plan: NVMe Migration for Longhorn Storage

**Branch**: `037-nvme-migration` | **Date**: 2026-04-04 | **Spec**: [spec.md](spec.md)

## Summary

Format and mount the 1.7TB NVMe drive (`nvme0n1`) on each cluster node at `/mnt/nvme`, add it to Longhorn as a schedulable disk, evict all replicas from the existing eMMC disk (`/var/lib/longhorn`), then remove the eMMC disk entry from Longhorn. Implemented as a new `nvme-prep` Ansible role plus a `playbooks/utilities/nvme-migrate.yml` utility playbook that orchestrates the full migration.

## Phase 1 Design

### Role: `nvme-prep`

**File**: `roles/nvme-prep/defaults/main.yml`

```yaml
nvme_device: /dev/nvme0n1
nvme_partition: /dev/nvme0n1p1
nvme_mount_path: /mnt/nvme
nvme_label: longhorn-nvme
nvme_mount_opts: defaults,noatime,nodiratime,nofail,x-systemd.device-timeout=30s
```

`nofail` is mandatory: if the NVMe fails to enumerate at boot (PCIe issue, connector problem), the node must continue booting so K3s and etcd remain up. Without `nofail`, a drive failure drops the node into emergency mode — on server nodes (node1, node3, node5) this risks etcd quorum loss. `x-systemd.device-timeout=30s` prevents systemd from hanging indefinitely waiting for the device.

**File**: `roles/nvme-prep/tasks/main.yml` — task sequence:

1. Install `e2fsprogs` and `parted`
2. Ensure `{{ nvme_mount_path }}` directory exists
3. `blkid` probe on `nvme_partition` → register `nvme_part_uuid`
4. Create GPT label on `nvme_device` (`when: nvme_part_uuid.stdout == ""`)
5. Create partition 1 spanning 0–100% (`when: nvme_part_uuid.stdout == ""`)
6. Format ext4 with label (`when: nvme_part_uuid.stdout == ""`, `force: false`)
7. Re-probe UUID after potential mkfs
8. Mount and persist fstab via `ansible.posix.mount` (`state: mounted`)
9. Disable PCIe ASPM in `/boot/firmware/cmdline.txt` (notify reboot handler)

**File**: `roles/nvme-prep/handlers/main.yml`

```yaml
- name: Reboot node
  ansible.builtin.reboot:
    reboot_timeout: 300
    post_reboot_delay: 30
```

The handler only fires if `cmdline.txt` was changed (first run). Subsequent runs are idempotent and do not trigger a reboot.

### Playbook: `playbooks/utilities/nvme-migrate.yml`

Four-play structure matching the four user stories. Designed to be run once per cluster, but individual plays can be re-run safely.

**Play 1 — NVMe Prep** (`hosts: cluster`):
- Include `nvme-prep` role on all 6 nodes

**Play 2 — Longhorn Disk Registration** (`hosts: servers[0]`, i.e., node1 only):
- For each node in `groups['cluster']` that is reachable:
  - `kubectl patch nodes.longhorn.io <node> --type merge` to add `nvme-disk` entry
  - Path: `/mnt/nvme`, `allowScheduling: true`, `evictionRequested: false`
  - `storageReserved`: variable `nvme_longhorn_reserved_bytes` (default 53687091200 = 50Gi)
- Wait for each new disk's `conditions.Schedulable = true` before proceeding

**Play 3 — Eviction** (`hosts: servers[0]`):
- For each node in `groups['cluster']`:
  - Read `spec.disks` to find the eMMC disk key (pattern `default-disk-*`)
  - Patch: `allowScheduling: false`, `evictionRequested: true` on eMMC disk
  - Poll `status.diskStatus.<key>.scheduledReplica | length` every 30 seconds
  - Timeout after 30 minutes per node (generous; 36Gi total at NVMe speeds)

**Play 4 — eMMC Disk Removal** (`hosts: servers[0]`):
- For each node:
  - Confirm `scheduledReplica` is empty (safety gate)
  - `kubectl patch --type json` `remove` op on eMMC disk key
  - Verify Longhorn node shows only `nvme-disk` in `spec.disks`

### Key implementation notes

- **eMMC disk key discovery**: The key follows `default-disk-<fsid>` and varies per node. The playbook must query it at runtime with `jq '.spec.disks | keys[] | select(startswith("default-disk"))'` rather than hardcoding.
- **Replica count safety**: With 2 replicas per volume and 6 nodes, evicting one node at a time is safe — the other replica remains available. However, do not evict multiple nodes simultaneously.
- **No downtime expected**: Longhorn replica rebuild is online. Attached PVCs remain accessible throughout migration.
- **eMMC cleanup after migration**: Once Longhorn disk removal (US4) is confirmed, Play 4 includes a cleanup task that removes `/var/lib/longhorn/replicas/` and `/var/lib/longhorn/longhorn-disk.cfg` from the eMMC on each node. The `/var/lib/longhorn/` directory itself is preserved (Longhorn's compiled-in default path; removing it entirely could cause issues on cluster rebuild). This reclaims the 11–29GB of replica data currently occupying eMMC per node.
- **`nofail` mount**: Longhorn will mark the disk `not-ready` if the NVMe is absent at runtime (e.g., after a failed boot where the drive didn't enumerate). This is the correct failure mode — volumes rebuild on surviving nodes rather than the entire node disappearing from the cluster.
